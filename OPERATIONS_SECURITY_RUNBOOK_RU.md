# Pelby: Защита данных и операционный регламент

Этот документ фиксирует минимально безопасный режим, чтобы не потерять пользовательские данные.

## 1. Как сейчас устроено хранение

1. На устройстве пользователя данные сохраняются в `AsyncStorage` (локальная резервная копия).
2. Те же данные синхронизируются на backend в Postgres через `/sync/push`.
3. Админка читает эти серверные снапшоты из `/admin/*`.

Итог: каждая запись хранится дублем:
- локально на телефоне пользователя;
- на сервере в Postgres (видно в админке).

## 2. Критичные настройки Render (обязательно)

В Web Service `horeca-backend` в `Environment` должны быть:

1. `DATABASE_URL` = **Internal Database URL** вашего Render Postgres.
2. `ADMIN_EMAIL` = уникальный email админа.
3. `ADMIN_PASSWORD` = длинный уникальный пароль (не меньше 20 символов).
4. `CORS_ORIGIN` = `*` (или список доменов через запятую).
5. (Опционально) `ADMIN_LOGIN_MAX_ATTEMPTS=30`, `ADMIN_LOGIN_WINDOW_MS=900000`.

Важно:
- если `ADMIN_EMAIL`/`ADMIN_PASSWORD` не заданы, `/admin/*` отключается (503);
- дефолтные значения нельзя использовать.

## 3. Обязательный бэкап (каждый день)

Экспорт серверных снапшотов:

```bash
API_BASE_URL="https://horeca-backend-6zl1.onrender.com" \
ADMIN_EMAIL="..." \
ADMIN_PASSWORD="..." \
npm run backup:server
```

Результат: папка `.backups/server-json/snapshot_<timestamp>/` с файлами всех пользователей.

Минимум:
1. Делать экспорт ежедневно.
2. Копировать архив/папку в отдельное хранилище (Google Drive/S3/внешний диск).

## 4. Smoke-проверка после каждого деплоя

```bash
API_BASE_URL="https://horeca-backend-6zl1.onrender.com" \
ADMIN_EMAIL="..." \
ADMIN_PASSWORD="..." \
npm run smoke:server
```

Проверяет:
1. `/health`
2. `/health/db`
3. `sync/push -> sync/pull`
4. `admin/users`, `admin/user/:id` (если заданы админ-креды).

## 5. Ротация секретов (сделать сразу)

Уже засвеченные секреты нужно заменить:
1. GitHub PAT.
2. Пароль Postgres / строку подключения.
3. `ADMIN_PASSWORD`.

После ротации:
1. Обновить секреты в Render.
2. Сделать redeploy backend.
3. Прогнать `npm run smoke:server`.

## 6. Как не потерять данные при новой/пустой БД

В приложении включен защитный сценарий:
1. Если для пользователя серверный снапшот отсутствует (`404`), клиент делает принудительный push локальных данных.
2. Синк запускается:
   - на старте;
   - по таймеру;
   - при смене состояния приложения;
   - после изменений `AsyncStorage`.

Это нужно, чтобы данные с телефона автоматически восстанавливали новую БД.

## 7. Еженедельный чек

1. `npm run smoke:server` -> SUCCESS.
2. В админке открывается список пользователей.
3. Есть свежий backup за последние 24 часа.
4. В Render Postgres не `Free` и хватает storage.
5. В логах нет регулярных `sync/push error` / `server_error`.
